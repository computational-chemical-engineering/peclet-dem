# Solver Implementation Details

This document describes the mathematical models and numerical methods currently implemented in the codebase for simulating particle dynamics.

## 1. Time Integration Scheme

The simulation uses a **Predictor-Corrector** scheme, closely related to **Extended Position Based Dynamics (XPBD)**, but with a specific pipeline.

### Step 1: Prediction (Integration)
*File: `src/cuda/integration.cu` (`predict_and_clear_kernel`)*

The solver starts by predicting the state of particles based on external forces (Gravity) only.
Given state $\mathbf{x}_n, \mathbf{v}_n$, and time step $\Delta t$:

1.  **Velocity Prediction:**
    $$ \mathbf{v}_{pred} = \mathbf{v}_n + \mathbf{g} \Delta t $$

2.  **Position Prediction:**
    $$ \mathbf{x}_{pred} = \mathbf{x}_n + \mathbf{v}_{pred} \Delta t $$

3.  **Orientation Prediction:**
    $$ \mathbf{q}_{pred} = \mathbf{q}_n + \frac{1}{2} \Delta t \boldsymbol{\omega}_n \mathbf{q}_n $$
    *(Normalized per step)*

### Step 2: Ghost Generation (Periodic Boundaries)
*File: `src/cuda/periodicity.cu`* / *`src/cuda/broadphase.cu`*

If periodicity is enabled, "Ghost" particles are generated by copying real particles that are within a interaction margin of the domain boundaries. These ghosts serve as boundary conditions for collisions but are not integrated themselves.

## 2. Collision Detection

### Broadphase
*File: `src/cuda/broadphase.cu`*
*   Uses a Linear BVH (LBVH) based on Morton Codes (Z-order curve) for spatial sorting.
*   Traverses the tree to find AABB overlaps (Potential Collisions).

### Narrowphase
*File: `src/cuda/narrowphase.cu`*
*   Verifies potential collisions using Signed Distance Fields (SDF) or analytic geometry.
*   **Contact Constraint Generation:**
    *   Calculates overlap distance: $d_{eff} = \text{SDF}(\mathbf{x}) - \text{radius}$.
    *   If $d_{eff} < \text{safety\_margin}$, a contact is generated.
    *   Normal $\mathbf{n}$ is computed via SDF gradient.

### Decoupling Note
**Contact Detection is strictly decoupled from the Position Solver.**
*   The `narrowphase` logic runs **once** at the beginning of the time step to generate a fixed list of contact pairs.
*   The solver iterates on this fixed list. It does **not** detect new contacts (e.g., from tunneling during iterations), but it **does** re-evaluate the geometry (distance and normal) for the existing pairs based on the updated predicted positions at each iteration.


## 3. Position Solving

*File: `src/cuda/solver_position.cu`*

The solver resolves geometric overlaps using an **iterative Projected Jacobi** method. This implies that all constraints are evaluated in parallel, and their resulting position corrections are averaged.

**Method:** Extended Position Based Dynamics (XPBD) (Jacobi variant).

### Constraint Equation
For a contact between particle $A$ and particle $B$ (or static wall):
$$ C(\mathbf{x}) = |\mathbf{x}_A - \mathbf{x}_B| - (r_A + r_B) \ge 0 $$
*(implemented as evaluating current overlap vs target rest length)*

### Constraint Deltas
The Lagrange multiplier update $\Delta \lambda$ is computed as:
$$ \Delta \lambda = \frac{-C(\mathbf{x}_{pred})}{w_{total} + \tilde{\alpha}} $$

Where:
*   $w_{total} = w_A + w_B$ is the generalized inverse mass (including rotational inertia contribution).
    $$ w_i = m_i^{-1} + (\mathbf{r}_i \times \mathbf{n})^T \mathbf{I}_i^{-1} (\mathbf{r}_i \times \mathbf{n}) $$
*   $\tilde{\alpha} = \alpha / \Delta t^2$ is the compliance (set to 0 for rigid contacts).

### Updates
Position and Quaternion corrections are computed:
*   **Position:** $\Delta \mathbf{x} = w \Delta \lambda \mathbf{n}$
*   **Rotation:** $\Delta \mathbf{q} \approx \frac{1}{2} (\mathbf{I}^{-1} (\mathbf{r} \times \mathbf{n}) \Delta \lambda) \mathbf{q}$

### Averaging (Jacobi)
*File: `src/cuda/integration.cu` (`apply_updates_kernel`)*

Since a particle may be part of multiple constraints, the total correction acts as an average:
$$ \mathbf{x}_{pred}^{new} = \mathbf{x}_{pred} + \frac{1}{N} \sum \Delta \mathbf{x}_j $$
This process is repeated for `position_iterations` count.

## 4. Velocity Solving (Unused)

*File: `src/cuda/solver_velocity.cu`*

**Status:** The velocity solver logic exists in the codebase but is **NOT** currently called in the main `Simulation::step` loop (in `src/simulation.cpp`). This implies that explicit friction and restitution handling via velocity impulses is currently disabled.

**Implemented Logic:**
The unused kernel `solve_velocity_jacobi_kernel` implements a standard impulse-based velocity correction:

1.  **Relative Velocity:**
    $$ \mathbf{v}_{rel} = (\mathbf{v}_A + \boldsymbol{\omega}_A \times \mathbf{r}_A) - (\mathbf{v}_B + \boldsymbol{\omega}_B \times \mathbf{r}_B) $$

2.  **Effective Mass:**
    Computes generalized inverse mass $w_{total}$ similar to position solve.

3.  **Normal Restitution:**
    *   Calculates target normal velocity: $v_n^{target} = -e \cdot v_n^{old}$
        *   *Note:* Reads `d_vel` (start of step state) as "old" velocity.
        *   Clamps target to $0$ if $|v_n^{old}| < 0.1$ (stabilization threshold).
    *   Computes impulse $\lambda_n$ to achieve $v_n^{target}$.
    *   Clamps $\lambda_n \ge 0$ (repulsive only).

4.  **Friction (Tangential):**
    *   Computes tangential velocity $\mathbf{v}_t$.
    *   Computes impulse $\lambda_t$ to stop sliding ($\mathbf{v}_t \to 0$).
    *   **Coulomb Clamp:** Limits friction impulse: $|\lambda_t| \le \mu \lambda_n$.

5.  **Updates:**
    Accumulates velocity changes into `d_delta_vel` and `d_delta_ang_vel` using `atomicAdd`.

## 5. Final Update

*File: `src/cuda/integration.cu` (`final_commit_kernel`)*

The resolved predicted positions are committed as the new state for the next step.

$$ \mathbf{x}_{n+1} = \mathbf{x}_{pred}^{resolved} $$

**Note on Velocity Update:**
In standard XPBD, the velocity is usually updated as $\mathbf{v}_{n+1} = (\mathbf{x}_{n+1} - \mathbf{x}_n) / \Delta t$.
**Currently**, the code sets:
$$ \mathbf{v}_{n+1} = \mathbf{v}_{pred}^{resolved} $$
Since `solver_position` does not update velocity accumulators (`d_delta_vel`), and `solver_velocity` is not called, **velocity is effectively only updated by Gravity** ($\mathbf{v}_{new} \approx \mathbf{v}_{old} + \mathbf{g} \Delta t$). The position correction does not currently feed back into the velocity state in the committing step.
